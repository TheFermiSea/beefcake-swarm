# Phase 1: State Machine & Actor-Based Orchestration

## Inspiration Context
In Symphony, `workflow.ex` and `orchestrator.ex` manage the lifecycle of a task strictly. An agent cannot simply "do whatever"; it must transition through defined states: `Planning -> Execution -> Verification -> Review -> Handoff`.

## Agent Instructions

### 1. Define the Global State Machine
Create `coordination/src/workflow/state.rs`. Define a strict Rust `enum` for the workflow states.

```rust
#[derive(Debug, Clone, Serialize, Deserialize)]
pub enum WorkflowState {
    Initialized,
    Planning { context_packed: bool },
    Delegation { active_agents: Vec<AgentId> },
    Execution(AgentId),
    Verification(AgentId),
    DebateConsensus { participants: Vec<AgentId> },
    HandoffReady,
    Failed(String),
}
```

### 2. Refactor `orchestrator.rs` into an Actor
Modify `crates/swarm-agents/src/orchestrator.rs` or `coordination/src/router/orchestrator.rs` to run as a Tokio task that manages this state machine.

- It should hold a `tokio::sync::mpsc::Receiver<SwarmEvent>`.
- When an agent finishes a subtask, it sends an event (`SwarmEvent::TaskCompleted(AgentId, Result)`).
- The Orchestrator handles the event, transitions the state (e.g., `Execution -> Verification`), and sends a command channel message to the Verification agent.

### 3. Rig Integration
Use Rig's `AgentBuilder` dynamically based on the current state.

- During the Planning state, build a Manager agent.
- During the Execution state, build a Coder agent, feeding it the output generated by the Manager in the previous state.
- Ensure all Rig LLM calls are wrapped in `tracing::instrument` for OpenTelemetry capture.

### 4. Acceptance Criteria

- The orchestrator no longer uses a simple `while` loop calling agents sequentially.
- Agents run on independent Tokio spawned tasks.
- State transitions are rigorously logged and strictly enforced (e.g., an Execution agent cannot transition the system directly to Handoff; it must go to Verification).
